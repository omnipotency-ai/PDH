## Surprising Findings

- **extractInsightData.ts imports from `../src/types/domain`**: This is a boundary violation. Convex server code should not import from the React frontend `src/` directory. The `StructuredFoodAssessment` type should be moved to `shared/` or duplicated in convex validators. This is a `type`-only import so it has no runtime impact, but it creates a fragile coupling.

- **logs.ts is a 2018-line mega-file** that owns four distinct domains: log CRUD, profile management, data backup/restore, and backfill migrations. This is the largest file in convex/ by a wide margin and is a clear decomposition candidate.

- **foodLlmMatching.ts is effectively dead code at runtime**: The `matchUnresolvedItems` action is stubbed to return `{ matched: 0, unresolved: 0 }` without calling OpenAI. The file contains ~350 lines of functional prompt building and response parsing code that is only exercised by tests. The comment says "applyLlmResults is not yet implemented" and that the server pipeline now handles LLM matching directly (in foodParsing.ts).

- **Re-exports from logs.ts are orphaned**: `habitConfigValidator` and `habitTypeValidator` are re-exported with a comment saying "for use in sync.ts and other files" but no file imports them via the logs.ts path. The comment references a file (`sync.ts`) that uses the convex API layer, not direct imports.

- **sanitizeStringArray in convex/lib/inputSafety.ts has zero consumers** in the convex directory. The src/lib/inputSafety.ts has its own separate copy. This is a dead export.

- **Multiple files define their own `normalizeCanonicalName` function**: foodAssessments.ts, ingredientExposures.ts, ingredientOverrides.ts, and aggregateQueries.ts all define a local `normalizeCanonicalName(value: string): string` function with identical logic (`canonicalizeKnownFoodName(value) ?? normalizeFoodName(value)`). This is duplicated in 4 files. The `shared/foodCanonicalName.ts` module exports the same function and is already used by computeAggregates.ts and aggregateQueries.ts -- the other 3 files should use it too.

- **testFixtures.ts is registered in the Convex API** (visible in \_generated/api.d.ts). Convex treats any .ts file in the convex/ directory as a module. Since testFixtures.ts has no mutations/queries/actions, it has no runtime impact, but it is somewhat unusual to have test support files in the Convex module tree.

- **migrations.ts has no consumers via import**: All 8 exported mutations are Convex entry points (callable via dashboard or CLI). No other convex file imports from migrations.ts. This is correct for migration code.

- **foodLibrary.ts mergeDuplicates is ~500 LOC**: A single mutation handler that touches 7 different tables and performs full food trial summary rebuilds. This is the most complex single mutation in the codebase and would benefit from extraction.

- **computeAggregates.ts duplicates the assessment-to-evidence mapping logic** found in foodLibrary.ts (the `mapAssessmentsToRecords` pattern). Both files build `FoodEvidenceLog[]` and `normalizeAssessmentRecord()` arrays with near-identical code.
