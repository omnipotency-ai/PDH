## Surprising Findings

1. **foodCanonicalName.ts is barely used and duplicated.** Only 2 convex files import it; 3 other convex files (`foodAssessments.ts`, `ingredientOverrides.ts`, `ingredientExposures.ts`) define an identical local `normalizeCanonicalName` function instead of importing the shared one. This is unnecessary duplication and should be consolidated.

2. **foodMatching.ts has many "internal-only" exports.** Functions like `stripFoodAccents`, `findExactAliasCandidate`, `buildBucketOptions`, `splitMealIntoFoodPhrases` are exported but only consumed internally. These could be un-exported to tighten the module's public API, unless they are intentionally exported for future consumers or testing.

3. **foodEvidence.ts exports several types/constants that have no external consumer.** `TransitResolverPolicy`, `FoodTrialResolutionMode`, `CLINICAL_TRANSIT_RESOLVER_POLICY`, and `FoodEvidenceTrial` are all exported but only used internally. These appear to be exported "just in case" rather than for actual consumers.

4. **prefersSummaryCandidate in foodNormalize.ts is misplaced.** This function compares summary rows for deduplication and has nothing to do with food name normalization. It was likely added here because both convex/computeAggregates.ts and convex/aggregateQueries.ts needed it, but it should live in its own utility file for clarity.

5. **domain.ts re-exports shared/foodTypes.ts types.** This creates a dual-import path: consumers can import `FoodPrimaryStatus` from either `@shared/foodTypes` or `@/types/domain`. The comment says this is for "backward compatibility" but it means imports are inconsistent across the codebase.

6. **FoodCategory and FoodSubcategory types are defined but never used outside foodRegistry.ts.** These are fields on every registry entry but no code reads or filters by them. They may be aspirational (for future categorization features) or vestigial.

7. **BRAT_FOOD_KEYS (Set) is exported but no consumer uses it.** Only `BRAT_BASELINE_CANONICALS` (the array) is actually imported by foodSafetyUtils.ts. The Set wrapper appears to be dead code.

8. **ConfidenceRoute type is not imported by any file.** It's only used as a return type within foodMatching.ts. External consumers (convex/foodParsing.ts) use the result structurally without importing the type.

9. **foodRegistry.ts at 4,058 lines is the largest file in the entire project.** However, ~3,800 lines are food entry data (not logic), making it a data file that happens to be TypeScript. The actual logic footprint is ~250 lines.

10. **domain.ts imports from 4 different src/lib/ modules** (`aiModels`, `habitTemplates`, `streaks`, `units`). This creates a coupling chain that prevents domain.ts from being shared with the server. The types that need server-side access have already been correctly extracted to shared/foodTypes.ts.
