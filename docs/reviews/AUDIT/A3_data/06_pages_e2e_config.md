## Surprising Findings

- **`next-themes` is a phantom dependency.** It is listed in `dependencies` but never imported. The app has a fully custom `ThemeProvider` at `src/components/theme-provider.tsx`. This adds unnecessary weight to `node_modules` and could cause confusion.

- **`@clerk/backend` is unused.** Listed as a devDependency but never imported. Possibly a leftover from early Clerk setup or a former peer dependency of `@clerk/testing`.

- **Track.tsx is imported eagerly, not lazily.** All other pages use `lazy()` in `routeTree.tsx`, but `TrackPage` is imported directly (`import TrackPage from "./pages/Track"`). Since Track is the default route (`/`), this is likely intentional for faster initial load, but it means all 669 lines of Track.tsx (and its deep dependency tree) are in the main bundle.

- **`routeTree.tsx` is a "god file" at 525 lines.** It contains 5 non-trivial components (GlobalHeader, AppLayout, AuthLoadingFallback, LegacyMigration, RouteErrorBoundary) alongside route definitions. This is the highest-priority decomposition target.

- **`Patterns.tsx` has 2 inline components** (`DatabaseTabContent` at ~260 lines and `TodayLabel` at ~23 lines) that could be extracted to reduce file complexity.

- **`food-pipeline.spec.ts` at 1,221 lines** is the largest file in scope. It is a comprehensive adversarial test suite with 10 sections. While test files benefit from colocation, the shared selectors and helper functions at the top (~140 lines) could be extracted to a shared helpers file.

- **All 4 destructive-habits tests are SKIPPED.** They require "Cigarettes" habit which is not in the default template keys. These tests need setup logic to first add the custom habit.

- **`dotenv` is a production dependency but only used in `e2e/auth.setup.ts`.** It should be moved to `devDependencies`.

- **`store.ts` exports far more than just the Zustand store.** It also exports 7 type guards, 4 constants, 3 re-exported types, and 1 interface. This file serves as a "barrel" for domain utilities -- while functional, it blurs the boundary between state management and data model definitions.

- **`src/vite-env.d.ts` at 2 lines** is purely ambient type references and not a "real" module. It exists for TypeScript to recognize Vite/PWA virtual module types.
