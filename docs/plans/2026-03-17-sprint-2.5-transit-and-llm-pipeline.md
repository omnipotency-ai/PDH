## Wave 3: LLM Pipeline

**Goal:** Make the LLM food matching pipeline work end-to-end using the user's BYOK API key.

### Task 3.1: Architecture Decision — Client-Initiated BYOK

The pipeline should be **client-initiated BYOK**:

- User's API key is stored in IndexedDB (already implemented)
- Client detects unresolved foods after server pipeline runs
- Client calls a Convex action passing the API key transiently
- Action calls OpenAI, returns results
- Client applies results via Convex mutation

This preserves the BYOK model (user pays for their own API calls) and keeps the key off the server at rest.

### Task 3.2: Implement `matchUnresolvedItems` Action

**Files:**

- Modify: `convex/foodLlmMatching.ts` — un-stub the action

Implement the action:

- Accept: `apiKey`, `logId`, `unresolvedItems[]`
- Add `requireAuth(ctx)` (WQ-002)
- Build a focused prompt: "Given these food items that couldn't be matched deterministically, identify what they are"
- Call OpenAI (gpt-4o-mini by default — cheapest, fast, good enough for food matching)
- Parse structured response (JSON mode)
- Return: matched items with canonical names, confidence scores, reasoning

### Task 3.3: Implement `applyLlmResults` Mutation

**Files:**

- Modify: `convex/foodLlmMatching.ts` or `convex/foodParsing.ts`

Take the LLM results and apply them:

- Update the food log items with canonical names and `resolvedBy: "llm"`
- Create/update `ingredientExposures` for newly resolved items
- Schedule evidence reprocessing

### Task 3.4: Build the LLM Prompt

**Files:**

- Create: `shared/foodLlmPrompt.ts` or inline in the action

The prompt should:

- Include the food registry canonical list (or a relevant subset) so the LLM can match to known foods
- Handle multi-ingredient inputs ("chicken pasta with cheese sauce")
- Return structured JSON with: canonicalName, confidence, reasoning, suggestedZone
- Include web search instruction if the model supports it (for truly unknown foods)
- Be token-efficient (the user is paying)

### Task 3.5: Wire the Client Hook

**Files:**

- Modify or recreate: `src/hooks/useFoodLlmMatching.ts`
- Modify: `src/components/track/FoodMatchingModal.tsx` or Track.tsx

Wire the BYOK flow:

- After food log is processed by server pipeline, check for unresolved items
- If unresolved items exist AND user has an API key configured → auto-trigger LLM matching
- Show progress in the UI (matching indicator on unresolved items)
- On completion, update the food log items

### Task 3.6: Tests + Commit

Write tests for:

- LLM prompt construction (unit test — no API call needed)
- `applyLlmResults` mutation (Convex test)
- Result parsing (structured JSON → ProcessedFoodItem mapping)
- Error handling: invalid API key, rate limit, malformed response

Run `bun run test:unit` + `bun run typecheck`. Commit.
