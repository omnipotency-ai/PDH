### Wave 3B: Populate & Extend foodEmbeddings (REVISED — was "remove", now "activate")

**Rationale (revised 2026-03-18):** Embeddings solve the "Chelitos problem" — brand names, regional foods, and voice-to-text spelling variations that fuzzy matching can't handle. Embeddings are 1000-10000x cheaper than LLM calls per food lookup ($0.000001 vs $0.001-0.01). The infrastructure already exists in code; it just needs to be seeded and extended.

**The Chelitos Problem:**

- User voice-logs "Chelitos" (a Spanish toasted corn snack)
- Not in registry → fuzzy fails → LLM fallback (expensive, fragile)
- User manually picks "crispy crackers"
- Next day, voice says "Kaelitas" (different spelling) → whole cycle repeats
- With embeddings: "Kaelitas" embeds close to "Chelitos" → auto-resolves

**Implementation:**

1. **Seed registry embeddings:** Run `ensureFoodEmbeddings()` (already written in foodParsing.ts) to embed all ~150 registry entries using `text-embedding-3-small`. One-time cost: ~$0.0001.

2. **Extend schema for alias embeddings:** Add an `isAlias` flag or `aliasOf` field to `foodEmbeddings` so learned user aliases get embedded too:

   ```ts
   foodEmbeddings: defineTable({
     canonicalName: v.string(),
     sourceText: v.string(), // "chelitos" or "white rice"
     sourceType: v.union(v.literal("registry"), v.literal("alias")),
     zone,
     group,
     line,
     bucketKey,
     bucketLabel,
     embedding: v.array(v.float64()),
     embeddingSourceHash: v.optional(v.string()),
     updatedAt: v.number(),
   });
   ```

3. **On user alias creation:** When a user maps an unknown food to a canonical (via the matching modal), embed the alias text and insert into `foodEmbeddings` with `sourceType: "alias"`. This means future misspellings of that alias auto-resolve via vector similarity.

4. **Revised food matching pipeline priority:**

   ```
   1. Exact alias lookup (existing foodAliases table) → conf 0.99
   2. Deterministic canonicalization (registry examples) → conf 0.99
   3. Fuse.js fuzzy search → conf varies
   4. Embedding vector search (registry + aliases) → conf varies
   5. Merge fuzzy + embedding scores (65%/35%)
   6. Confidence routing: HIGH → auto, MEDIUM → pending, LOW → user picks
   7. LLM fallback ONLY for structurally ambiguous phrases with candidates
   ```

5. **LLM web search stays as last-resort fallback.** (Decision 2026-03-19: KEEP.) Web search is the fallback after embeddings/fuzzy/alias/LLM all fail. The matching chain: registry → fuzzy/alias → embeddings → LLM without web → LLM with web → "unknown food".

**Cost:** ~$0.0001 to seed, ~$0.000001 per new alias embedding. Essentially free.

**Estimated savings:** Eliminates most LLM fallback calls ($0.001-0.01 each). Self-improving: each user mapping makes future lookups better. Handles voice-to-text spelling variations naturally.

## Phase 4: LLM Context Pipeline Redesign

**Goal:** Replace bulk-log dumps to Dr. Poo with relevance-filtered, token-efficient context.

**Depends on:** Phase 2 (indexed queries). Can run in parallel with Phase 3.

### Wave 4A: Design the Ideal LLM Context Payload

**This is the McKinsey question.** What would a world-class data analyst design as the minimum sufficient context for Dr. Poo to do its job?

**Current approach (brute force):**

- Send ALL logs from the last N hours as raw JSON
- ~5-15KB of log data per call
- Most of it is irrelevant (stable safe foods, repeated habits)

**Proposed approach (strategic):**

**Key design decision: variable time windows by log type and surgery.**

Different log types have different relevance horizons. Food logs need the longest window
(because transit time determines when food affects BMs), while habits and fluids are
immediate-effect and only need recent context. The food window varies by surgery type
because colostomy patients have slower transit than ileostomy patients.

| Log type    | Window                        | Rationale                                        |
| ----------- | ----------------------------- | ------------------------------------------------ |
| Food        | Surgery-dependent (see below) | Transit time determines relevance                |
| BMs         | 48h always                    | Pattern context (frequency, trend)               |
| Habits      | 24h always                    | Immediate behavioral context                     |
| Fluids      | 24h always                    | Hydration is immediate-effect                    |
| Activities  | 24h always                    | Exercise, movement — immediate context           |
| Medications | 24h always                    | Compliance, dosing — immediate context           |
| All others  | 24h always                    | Default window for any log type not listed above |

**Food log window by surgery type:**

| Surgery type       | Food window | Rationale                                                       |
| ------------------ | ----------- | --------------------------------------------------------------- |
| Ileostomy reversal | 48h         | Faster transit, most food through in 24-36h                     |
| Colostomy reversal | 72h         | Slower transit, constipation-prone                              |
| + slow modifiers   | 96h         | Pregnancy, opioids, iron supplements slow transit significantly |

Slow transit modifiers are already in the health profile (medications, pregnancy status).
`buildRecentEvents()` reads the profile and adjusts the food window dynamically.

For foods beyond the event window that are still potentially in transit (e.g., a food
eaten 80h ago with estimated transit 72-96h due to modifiers), the `foodContext` section
includes a `stillInTransit` list computed from the transit model — no raw logs needed.

```typescript
LLM Context Payload Structure:
{
  // 1. Patient snapshot (slowly changing, ~200 tokens)
  patient: {
    daysSinceReversal: 32,
    surgeryType: "ileostomy_reversal",
    medications: ["loperamide_prn"],
    baselineTransitMinutes: 420,
    currentBristolTrend: "improving (6.3 -> 5.4 -> 4.5 weekly avg)",
  },

  // 2. Recent events (variable windows by log type, ~500 tokens)
  //    Food window: 48h (ileo) / 72h (colo) / 96h (colo + slow modifiers)
  //    BM window: 48h always
  //    Habits/fluids window: 24h always
  recentEvents: {
    bowelMovements: [  // 48h window
      { time: "2h ago", bristol: 4, notes: "firm, easy", volume: "medium" },
      { time: "8h ago", bristol: 3, notes: "hard, small" },
    ],
    foodsEaten: [  // surgery-dependent window
      { time: "4h ago", foods: ["white rice", "boiled chicken"], zone: 1 },
      { time: "12h ago", foods: ["crispy crackers", "tea"], zone: 1 },
    ],
    habits: { cigarettes: 5, sleep: "5h fragmented" },  // 24h window
    fluids: { totalMl: 1200, breakdown: { water: 600, tea: 400, coffee: 200 } },  // 24h window
  },

  // 3. Delta signals (what changed, ~200 tokens)
  //    v1: concrete, computable deltas only. No pattern detection.
  deltas: {
    newFoodsThisWeek: ["sweet biscuit"],  // from knownFoods set
    bristolChangeFromYesterday: -0.5,     // simple math: today avg - yesterday avg
    recentCulpritExposure: null,          // from foodTrialSummary flags
    habitStreaks: { loperamide: 14 },     // consecutive days of compliance
  },

  // 4. Food intelligence (only relevant foods, ~300 tokens)
  foodContext: {
    activeFoodTrials: [
      { food: "sweet biscuit", status: "testing", exposures: 2, lastBristolAfter: 4 },
    ],
    recentSafe: ["white rice", "boiled chicken", "crispy crackers"],
    recentFlags: [],
    stillInTransit: [],  // foods beyond event window but still in estimated transit
    nextToTry: { food: "boiled egg", reason: "Zone 1, high protein, low residue" },
  },

  // 5. Conversation continuity (last 2-3 raw messages, ~300 tokens)
  //    Already summarized Wed+Sun — include raw messages between summaries
  conversationContext: "Patient asked about adding parsnips. Advised waiting for stable week.",

  // 6. User's message/trigger
  userMessage: "Just had a really satisfying poo, biggest one yet",
}
```

**Token comparison:**

- Current: ~3,000-5,000 tokens of raw logs
- Proposed: ~1,500 tokens of pre-computed, relevance-filtered context
- **50-70% token reduction per LLM call**
- Variable windows keep tokens tight: habits/fluids at 24h avoid bloating the payload

## Canonical Name Staleness: The Full Explanation

### Why it currently happens

The food registry (`shared/foodRegistry.ts`) maps canonical names like "white rice" with example arrays like `["rice", "white rice", "plain rice", "boiled rice"]`. When a user logs "boiled rice", the pipeline maps it to canonical "white rice" and stores `canonicalName: "white rice"` in multiple tables.

If the registry later changes — e.g., "white rice" is renamed to "steamed rice", or two entries merge — the stored `canonicalName` values become stale. The runtime code handles this by normalizing every read:

```ts
const canonicalName =
  canonicalizeKnownFoodName(row.canonicalName) ??
  normalizeFoodName(row.canonicalName);
```

This forces `.collect()` on every query (can't use indexes because stored values don't match current registry).

### Why it's a migration problem, not a runtime problem

**Registry updates only happen at planned release times.** New foods are added during specific update sessions, not continuously. This means:

1. Between releases, canonical names are stable and indexes work perfectly
2. At release time, run the migration script (Wave 1B) to normalize all stored values
3. After migration, indexes work again until the next registry update

### What the migration script does

```ts
// For each row in foodAssessments, ingredientExposures, foodTrialSummary, foodAliases:
const currentCanonical =
  canonicalizeKnownFoodName(row.canonicalName) ??
  normalizeFoodName(row.canonicalName);

if (currentCanonical !== row.canonicalName) {
  await ctx.db.patch(row._id, { canonicalName: currentCanonical });
}
```

This is idempotent, safe to re-run, and should be added to the registry update release checklist.

### What changes during a registry update

| Change Type           | Example                                                   | Effect on Stored Data                                 |
| --------------------- | --------------------------------------------------------- | ----------------------------------------------------- |
| New food added        | Add "boiled parsnips"                                     | No effect — new entries have no stored references yet |
| Example list expanded | Add "steamed rice" as alias for "white rice"              | No effect — canonical name unchanged                  |
| Canonical renamed     | "white toast" -> "toast (white)"                          | All stored "white toast" references become stale      |
| Entries merged        | "boiled carrots" + "boiled parsnips" -> "boiled root veg" | Stored references to old names become stale           |
| Entry split           | "mixed salad" -> "green salad" + "bean salad"             | Ambiguous — needs manual mapping in migration         |

**Key insight:** Adding new foods (the most common update) causes ZERO staleness. Only renames and merges cause it, and those are rare.
